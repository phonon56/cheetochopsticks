What this is

This package is a replacement for the inaccessible GovOutreach third-party CORA submission portals currently linked from coloradosprings.gov/City-Clerk/CORA. It was built by two residents and is offered to the City at no charge.

Why this matters: The current GovOutreach portals fail WCAG 2.1 AA accessibility standards — they render empty content to screen readers and other assistive technology. The DOJ Final Rule compliance deadline for Colorado Springs (50,000+ population) was April 24, 2026. This package provides a compliant replacement that the City can deploy on existing infrastructure with no new vendor contracts.

The package consists of three deliverables:

  • cos-cora-form-v2.html — the accessible public-facing form
  • cos-cora-azure-function.zip — the Azure Function backend that processes submissions
  • cos-cora-implementation-guide.html — this document

Tech stack confirmed

coloradosprings.gov runs Drupal — confirmed by the drupalSettings JavaScript object present on every page. Specifically it runs the Tyler Colorado CMS platform, a Drupal multi-site distribution managed by Tyler Technologies.

It is not .NET. No ASP.NET ViewState was detected. No .aspx links were found on the domain.

This is good news. Drupal has a mature, accessible, free form builder module (Webform) that can replace GovOutreach entirely without custom code. See the Native Drupal option section for that path.

The Azure Function path described in this guide works alongside Drupal — the form can be hosted as a static page or embedded in Drupal, with the Function handling backend processing independently of the CMS.

Form submission modes

The form automatically operates in one of two modes based on the CONFIG.AZURE_FUNCTION_URL setting at the top of the HTML file:

Mode 1 Azure Function (recommended)

When AZURE_FUNCTION_URL is set to a real endpoint, the form POSTs to the Azure Function. The function generates a reference number, sends a confirmation email to the resident, and routes the request to the correct department. A green banner shows at the top of the form confirming online submission is active.

Mode 2 Mailto fallback

When AZURE_FUNCTION_URL is empty (default until Azure is deployed), the form collects the resident's information and pre-fills a mailto: link with everything ready to send. The resident's email client opens with the full request pre-populated. A purple banner explains this to the resident.

Automatic fallback: Even in Azure mode, if the Azure Function returns an error or is unreachable, the form automatically falls back to mailto so the resident is never stranded. No submission is ever lost.

Configuring the form

Open cos-cora-form-v2.html in a text editor. Find the <script> block near the top (inside the <head>):

const CONFIG = {
  AZURE_FUNCTION_URL: '',
  // When deployed:
  // 'https://func-cora-cos-prod.azurewebsites.net/api/cora-submit?code=YOUR_KEY'

  CORA_EMAIL: 'City.Communications@coloradosprings.gov',
};

Two values to set:

  • AZURE_FUNCTION_URL — leave empty for mailto mode. After deploying the Azure Function (Step 4 below), paste the full endpoint URL here including the ?code= function key.
  • CORA_EMAIL — the email address that receives mailto fallback submissions. Update this to the actual CORA coordinator email.

That is the only configuration required in the HTML file. Everything else is handled by the Azure Function's environment variables.

Deploying the form

Three options, from fastest to most integrated:

Option A — Static file (fastest, works today)

Upload cos-cora-form-v2.html to the city's web server as a static file. Link to it from the CORA page. No Drupal required.

Option B — Drupal basic page

Create a new Basic Page in Drupal. Set the text format to Full HTML. Paste the entire HTML file contents as the page body. Save. The form will render within the Drupal template.

Option C — Replace GovOutreach links

On the existing CORA page in Drupal, replace each GovOutreach link with a direct link to the form with a pre-selected department parameter:

Contracting / Procurement — Submit a Request
href="/cora-request?dept=procurement"

Add a small script to the form page that reads the URL parameter and pre-selects the department dropdown on load:

const params = new URLSearchParams(window.location.search);
const dept = params.get('dept');
if (dept) {
  const sel = document.getElementById('department');
  if (sel) sel.value = dept;
}

Azure Function prerequisites

  • Azure subscription (the City's existing Azure tenant works)
  • Node.js 18 or higher (for local testing only)
  • Azure Functions Core Tools v4 (for local testing only)
  • A Microsoft 365 mailbox to use as the sending address

Install Core Tools for local development:

npm install -g azure-functions-core-tools@4 --unsafe-perm true

Unzip the function package:

unzip cos-cora-azure-function.zip
cd cos-cora-azure-function
npm install

Step 1 — Set up the sending mailbox

Create a dedicated Microsoft 365 mailbox for the function to send from:

cora-noreply@coloradosprings.gov

Generate an App Password for SMTP authentication:

  1. Sign in to the mailbox account

    Go to account.microsoft.com and sign in as cora-noreply@coloradosprings.gov.

  2. Navigate to App Passwords

    Security → Advanced security options → App passwords → Create new app password.

  3. Name it and copy the password

    Name: CORA Azure Function. Copy the generated password immediately — it will not be shown again. Store it in Azure Key Vault or the Azure Function's Application Settings (never in source code).

Why an App Password? Microsoft 365 accounts with MFA enabled block standard SMTP authentication. App Passwords bypass MFA for service accounts without disabling MFA on the account. This is the recommended Microsoft approach for SMTP relay from automated systems.

Step 2 — Deploy to Azure

Option A — Azure CLI (recommended for IT)

# Login to Azure
az login

# Create resource group
az group create \
  --name rg-cora-prod \
  --location eastus

# Create storage account (required by Functions)
az storage account create \
  --name stcoracosprod \
  --location eastus \
  --resource-group rg-cora-prod \
  --sku Standard_LRS

# Create Function App
az functionapp create \
  --resource-group rg-cora-prod \
  --consumption-plan-location eastus \
  --runtime node \
  --runtime-version 18 \
  --functions-version 4 \
  --name func-cora-cos-prod \
  --storage-account stcoracosprod

# Deploy the function code
cd cos-cora-azure-function
func azure functionapp publish func-cora-cos-prod

Option B — Azure Portal (no CLI)

  1. portal.azure.com → Create a resource → Function App
  2. Runtime: Node.js 18 · Region: East US · Plan: Consumption (Serverless)
  3. After creation: Functions → Add → HTTP trigger → Name: CoraSubmit
  4. Paste the contents of CoraSubmit/index.js into the code editor
  5. Replace function.json with the provided bindings

Option C — GitHub Actions CI/CD (best for ongoing maintenance)

Push the package to a private GitHub repo. In Azure portal: Function App → Deployment Center → GitHub → connect and save. Azure deploys automatically on every push to main.

Step 4 — Test locally before going live

# Copy the example settings and fill in real credentials
cp local.settings.json.example local.settings.json
# Edit local.settings.json with real SMTP credentials

# Start the function locally
func start

# In a second terminal, run the test
node test/test.js

A successful test produces:

  • A confirmation email at the requester-email test address
  • A routing email to the procurement coordinator email
  • A reference number printed in the terminal (format: CORA-YYYYMMDD-XXXXXXXX)

Check both email inboxes before proceeding to deployment.

Step 5 — Connect the form to the function

After deploying, get the function URL from Azure portal:

Function App → Functions → CoraSubmit → Get Function URL

The URL looks like:

https://func-cora-cos-prod.azurewebsites.net/api/cora-submit?code=ABC123def...

Open cos-cora-form-v2.html and update the CONFIG block:

const CONFIG = {
  AZURE_FUNCTION_URL: 'https://func-cora-cos-prod.azurewebsites.net/api/cora-submit?code=ABC123def...',
  CORA_EMAIL: 'City.Communications@coloradosprings.gov',
};

The form will now show the green "Online submission active" banner and POST directly to the Azure Function on submit.

Native Drupal option (Webform module)

coloradosprings.gov runs the Tyler Colorado Drupal CMS platform. The Drupal Webform module provides a native alternative to the Azure Function that lives entirely within the Drupal installation — no external infrastructure required.

Check if Webform is already installed

Log in to Drupal admin and navigate to:

/admin/structure/webform

If the page loads — Webform is installed. If you get a 404 — request installation from Tyler Colorado.

Request installation from Tyler Colorado

Email your Tyler Colorado account contact:

"We need the Drupal Webform module installed on coloradosprings.gov. It is available at drupal.org/project/webform and is one of the most-used contributed modules in the Drupal ecosystem. We need it to replace inaccessible third-party form vendors on our CORA page to meet DOJ Final Rule accessibility requirements. Installation: composer require drupal/webform && drush en webform"

Build the form in Drupal Webform

Once installed: Structure → Webforms → Add webform → Name: CORA Records Request

Add fields matching the form in this package. The Webform module handles all conditional logic, email routing, confirmation messages, submission logging, and WCAG 2.1 AA accessibility natively.

Email routing by department is handled through Webform's Handler system: Settings → Emails/Handlers → Add email. One handler per department, with a condition checking the department select field value.

Webform advantage: Submissions are stored in Drupal's database, giving city staff a searchable submission log directly in the Drupal admin. No Azure infrastructure required. No SMTP configuration required. Uses the City's existing Drupal mail configuration.
