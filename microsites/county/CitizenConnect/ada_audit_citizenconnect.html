Functional issues beyond WCAG

The ticket system loses residents' submissions

These problems don't violate accessibility rules, but they break the service for residents who rely on it. They're at least as urgent as the WCAG findings below.

  • No history of the original submission. After a service request is filed, the ticket view does not show the original details the resident submitted. There is no way to confirm what was sent, when, or against which address — which also makes it impossible to dispute a miscategorization.
  • Previous email correspondence can disappear. Follow-up emails between a resident and the county are not reliably attached to the ticket thread. Residents lose the audit trail of what was communicated, and staff can't easily pick up a case handed off from another department.
  • Pothole submission is broken. The pothole-reporting form does not complete submission. Residents receive no confirmation, and no ticket appears in the system. The defect is silent — users believe their report was filed when it was not.
Recommended fix: Replace or supplement the automated ticket system with a monitored intake inbox staffed by a real person who routes each request to the right department. Email preserves the full thread automatically, gives residents a durable receipt, and re-introduces a human who can read the first sentence of a broken pothole report and still get it to Public Works.

Serious — fix before launch

Serious
Search input has no accessible label
The "Check Status" service-request search field (#sr_number) has no <label>, no aria-label, and no aria-labelledby. Only a placeholder is present, which disappears when the user starts typing and is never read reliably by screen readers.
Fix: Add <label for="sr_number">Service request number</label> — visually hide it with a .sr-only class if needed — or add aria-label="Service request number" directly on the input.

Moderate — fix soon

Moderate
Mobile menu toggle is keyboard-inaccessible
The hamburger menu uses a CSS checkbox-hack: a real <input type="checkbox" id="showmenu"> set to display:none. Because it is hidden via CSS, it is removed from the accessibility tree entirely. Keyboard and screen-reader users cannot open the mobile navigation.
Fix: Replace with a <button> that toggles the nav with JS. Apply aria-expanded and aria-controls to the button, and manage focus into the menu on open.

Moderate
Icon images have empty alt text with no decorative role
Four homepage icons (icon1–icon4.png) have alt="" but are not marked role="presentation" or aria-hidden="true", and none sit inside a link or labeled container that would provide context. If they convey service categories they need descriptive alt text; if they are purely decorative, they need explicit presentational role.
Fix: Add alt="Service requests" / etc. if meaningful, or role="presentation" if purely decorative. Also add aria-hidden="true" to the surrounding SVG icons.

Three news thumbnails (NR-WEB-IMAGE-TEMPLATE-6.png, Vote-NR-Image.png, NR-WEB-IMAGE-TEMPLATE-3-1.png) have alt="". They appear to be informative editorial photos that provide context for each article — making them non-decorative and requiring meaningful descriptions.

Minor / best practice

Minor
Positive tabindex values disrupt tab order
The service-request input has tabindex="5" and the "Check Status" button has tabindex="6". Positive tabindex values pull elements to the front of the tab sequence regardless of DOM order, which can confuse keyboard users navigating the rest of the page.
Fix: Remove explicit tabindex values and rely on natural DOM order. Use CSS to reorder visual presentation if needed.
2.4.3
Minor
Multiple navigation landmarks have no distinguishing labels
The page contains several <nav> elements (main nav, footer links, etc.) with no aria-label or aria-labelledby. Screen reader users who browse by landmarks will hear repeated announcements of "navigation" with no way to differentiate them.
Fix: Add aria-label="Main", aria-label="Footer", etc. to each nav element.
2.4.1
Minor
article elements use inappropriate role="listitem"
Three news <article> elements in the "In the News" grid have role="listitem". The listitem role is not permitted on article elements per ARIA spec. This flag is marked "incomplete" by axe because these elements are currently hidden, but the pattern will cause issues when visible.
Fix: Remove the role="listitem" attribute from article elements. If you need list semantics, wrap items in a <ul> with <li> elements instead of articles.
4.1.2
Minor
Third-party iframe embeds need verification
Two embedded iframes (YouTube widget via woxo.tech, Twitter timeline) are present. The YouTube widget title reads "Youtube widgets for website" — generic. The Twitter iframe titled "Twitter settings iframe" is a hidden utility frame, but if the Twitter timeline widget itself is present it requires a meaningful title. Third-party widgets often have significant internal accessibility issues.
Fix: Give the YouTube embed a descriptive title (title="El Paso County video highlights"). Audit third-party widgets independently or replace with natively accessible alternatives.
4.1.2
Minor
Page title is not sufficiently descriptive
The page title is "El Paso County, Colorado" — identical to the root domain's title. In multi-tab workflows, users cannot distinguish this tab from others. A more descriptive title aids screen reader orientation and browser history scanning.
Fix: Change to something like "Citizen Connect — El Paso County, Colorado" to uniquely identify the portal.
