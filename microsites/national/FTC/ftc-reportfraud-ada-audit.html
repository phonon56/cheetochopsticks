Executive Summary

This audit examined the complete end-to-end user flow of reportfraud.ftc.gov — the federal government's primary interface for citizens to report fraud, scams, and deceptive business practices. The audit covered three URL paths: the category-selection wizard (/assistant), the personal information form (/form/main), and the review/confirmation screen (/form/confirmation).

A total of 19 distinct accessibility failures were identified across the flow, including 6 critical violations and 6 serious violations. The site currently lacks a Section 508 accessibility contact link anywhere in its interface, which is itself a compliance gap under federal accessibility law. Multiple failures compound each other: the combination of no required-field marking, no inline error feedback, and no focus management on submission means that a screen reader user who makes an error has no reliable path to identify or correct it.

Scope & Methodology

The audit was conducted by programmatic DOM inspection using the browser's accessibility tree, computed style analysis, and JavaScript evaluation of ARIA attributes, input associations, focus states, and keyboard operability across each step of the complete form flow. The following pages were audited in sequence:

  • Step 1 — reportfraud.ftc.gov/assistant — Category selection (wizard landing)
  • Step 2 — reportfraud.ftc.gov/assistant — Subcategory selection (dynamically injected)
  • Step 3 — reportfraud.ftc.gov/form/main — Personal information form
  • Step 4 — reportfraud.ftc.gov/form/confirmation — Review and submission

Automated browser extension tools (axe DevTools, WAVE, Lighthouse) were not relied upon as the primary method of detection. As documented in prior research, automated tools detect approximately 30–40% of WCAG failures. The remaining failures — particularly those involving context, focus management, dynamic content, and keyboard operability — require manual evaluation.

Critical Violations — WCAG 2.1 Failure

Critical violations constitute clear, testable WCAG 2.1 Level A or AA failures that directly prevent or substantially impede access for users of assistive technology.

Critical
No H1 heading on any page in the flow All pages
Affects: /assistant · /form/main · /form/confirmation

Every page in the flow — the wizard, the form, and the confirmation screen — contains zero H1 elements. Section headings are implemented as H3 throughout ("Is your report about any of these common problems?", "About you", "Your details"). There is no H1 anywhere in the application. Screen reader users who navigate by headings on page load receive no primary landmark identifying the page's purpose. The H3-only structure also breaks heading hierarchy, as there is no H2 level between the absent H1 and the H3 content headings.

document.querySelectorAll('h1').length → 0 (on all three pages) Headings found: H3 × 6, H5 × 1 (on /form/main) Heading levels used: H3 and H5 only — no H1 or H2

WCAG 1.3.1 Info and Relationships · 2.4.6 Headings and Labels

Recommended FixAdd a visually positioned (or visually-hidden) H1 to each page. The wizard H1 should identify the page purpose ("Report Fraud — Step 1 of 3"). The form H1 should read "Your Information." Existing H3 section headings should become H2 elements, maintaining a logical H1 → H2 → H3 hierarchy.
Critical
Radio groups have no fieldset or legend — question context lost /assistant · /form/main

The wizard's primary category radio group (10 options), its subcategory radio group (dynamically injected), and both yes/no radio groups on /form/main ("Are you filing this report for someone else?" and "Are you filing on behalf of your small business?") are all implemented without <fieldset> and <legend> elements.

The question text is rendered either as an H3 element or as a <label> element with no for attribute — meaning it is visually proximate to the radio buttons but programmatically disconnected. A screen reader user navigating by form controls will hear "Yes, radio button, 1 of 2" with no announcement of what they are saying yes to. On the wizard, they hear a category label with no announcement of the group question.

fieldset count: 0 (on all pages) Radio group "yesOrNoReportOther": question label has no for="" attribute Question text element: <label class="col-sm col-md-7 rf-p">Are you filing...</label> → Not wired to any input. Programmatically invisible as a form association.

WCAG 1.3.1 Info and Relationships · 3.3.2 Labels or Instructions

Recommended FixWrap each radio group in a <fieldset>. Move the question text into a <legend>. Screen readers will then announce: "Are you filing this report for someone else? Yes, radio button, 1 of 2."
Critical
No required field indicators — programmatic or visual /form/main

Zero inputs on the personal information form have the required attribute or aria-required="true". The form does enforce required fields on submission (a modal fires when the form is blank), but no per-field signal exists before or during completion. Screen readers, browser-native form validation, and keyboard-only users have no mechanism to identify which fields are mandatory before attempting to submit. This violates both the labeling requirement (users cannot know instructions apply to a field) and the error prevention requirement (no pre-submission cue exists).

inputs with required attribute: 0 inputs with aria-required="true": 0 inputs with aria-describedby: 0 Validation mechanism: modal dialog fires on submit ("Your report is blank")

WCAG 3.3.1 Error Identification · 3.3.2 Labels or Instructions

Recommended FixAdd required to all mandatory inputs. Add a visual indicator (asterisk with a key: "* Required field") and a <span aria-hidden="true">*</span> within each label so screen readers do not read the asterisk aloud. Alternatively use aria-required="true" and add a legend at the top of the form.
Critical
No visible focus outline on interactive elements All pages

Computed CSS styles confirm that the application removes the default browser focus outline on all button elements. The "Continue", "Back", and "Submit" buttons — the primary navigation controls for the entire flow — have outline: none applied. Keyboard-only users, switch-access users, and others who cannot use a pointer device cannot visually determine which element currently has focus at any point during form completion.

window.getComputedStyle(document.querySelector('button')).outline → "rgb(255, 255, 255) none 0px" outlineOffset: "0px" No :focus-visible override found in stylesheets.

WCAG 2.4.7 Focus Visible · 2.4.11 Focus Appearance (WCAG 2.2 AA)

Recommended FixRemove all outline: none / outline: 0 declarations from interactive elements. Replace with a custom focus style using :focus-visible: e.g., outline: 3px solid #0d2b55; outline-offset: 2px;. The FTC brand navy provides sufficient contrast against white backgrounds.
Critical
Dynamic content injection with no focus management /assistant step 2

When a category is selected on the wizard, a subcategory section ("Who were they pretending to be?") is dynamically injected into the page below the main radio group. Focus does not move to the new content — it remains on BODY. The subcategory heading has no tabindex attribute and cannot receive programmatic focus. A screen reader user selecting a category receives no announcement that new content has appeared and must discover it by continuing to navigate the page manually — which they may not do if they believe the page is complete.

After category selection: document.activeElement → BODY# (focus dropped to body) subcatHeading.tabindex → null (cannot receive focus) aria-live region update: none observed for new section

WCAG 4.1.3 Status Messages · 2.4.3 Focus Order · ARIA Live Region guidance

Recommended FixOn dynamic section injection, either: (a) move focus programmatically to the new section heading using heading.setAttribute('tabindex', '-1'); heading.focus();, or (b) announce the new section via an aria-live="polite" region. Option (a) is preferred as it also positions keyboard users at the correct location.
Critical
No Section 508 accessibility contact anywhere on the site All pages

No link to an accessibility statement, Section 508 compliance information, or an accessibility contact email exists anywhere on reportfraud.ftc.gov — not in the header, footer, or on any page in the flow. Under Section 508 of the Rehabilitation Act (29 U.S.C. § 794d) and the associated IT Accessibility Standards, federal agencies are required to provide a mechanism for users to report accessibility barriers and request accommodations. The FTC maintains an accessibility page at ftc.gov/accessibility, but it is not linked from this subdomain.

Search for /access|508|ada|disability/ across all links: 0 results mailto: links on any page: 0 Footer content: Privacy Policy, FAQs, Report Now, OMB Control# only

Section 508 · 36 C.F.R. Part 1194 · WCAG 2.4.4 Link Purpose

Recommended FixAdd a link in the footer to FTC's accessibility page (ftc.gov/accessibility) and include a direct mailto link for accessibility assistance. Suggested text: "Accessibility" linking to ftc.gov/accessibility, with supplemental text: "For accessibility assistance, email accessibility@ftc.gov."

Serious Violations

Serious
All personal data fields missing autocomplete tokens /form/main

Every personal information input on the form — first name, last name, address lines, city, state, zip, phone, email, age, military status — has no autocomplete attribute. WCAG 1.3.5 requires that inputs collecting personal data carry appropriate autocomplete tokens so that autofill tools (used heavily by users with motor impairments, cognitive disabilities, and users of switch access or eye gaze devices) can populate them without manual entry. Notably, Street Address 1 is explicitly set to autocomplete="disabled", which is not a valid value for this attribute (the correct value to disable autofill is "off"), suggesting intentional suppression of autofill on one of the most commonly autofilled fields.

rayfirstName autocomplete: null → should be: "given-name" raylastName autocomplete: null → should be: "family-name" address field 1 autocomplete: "disabled" (INVALID) → should be: "address-line1" rayaddresstwo autocomplete: null → should be: "address-line2" raycity autocomplete: null → should be: "address-level2" USZipCode autocomplete: null → should be: "postal-code" rayphone autocomplete: null → should be: "tel" rayemail autocomplete: null → should be: "email"

WCAG 1.3.5 Identify Input Purpose

Recommended FixAdd WCAG-compliant autocomplete token values to all personal data inputs as listed above. Remove autocomplete="disabled" from the address field — if autofill is intentionally blocked, use autocomplete="off", though suppressing autofill on a government form collecting personal data is itself an accessibility barrier and should be reconsidered.
Serious
No inline error feedback — errors announced via generic modal only /form/main

When the form is submitted with missing information, the application opens a modal dialog reading "Your report is blank. To file a report, you will need to fill in some information." No individual field receives aria-invalid="true". No field receives an aria-describedby association pointing to an error description. No inline error message appears adjacent to any field. A screen reader user who encounters this modal knows that something is missing but has no mechanism to identify which field(s) are problematic or where to navigate to correct them. This is compounded by the fact that fields are not marked required (Critical finding C3), so the user also had no pre-submission warning.

inputs with aria-invalid="true" after submit: 0 inputs with aria-describedby: 0 inline error elements found: 0 Error mechanism: modal dialog with generic message only Focus after error modal closes: returns to BODY (not to first error field)

WCAG 3.3.1 Error Identification · 3.3.3 Error Suggestion

Recommended FixOn submission failure: (1) Add aria-invalid="true" to each invalid field. (2) Inject inline error messages and associate them via aria-describedby. (3) Move focus to the first invalid field or to a summary error region at the top of the form. (4) When the error modal closes, focus should return to the first invalid field, not to BODY.
Serious
Skip link targets gov banner expand — not main content All pages

The only skip navigation link on every page in the flow targets #collapseFlag — the "Here's how you know" government banner toggle — rather than the page's main content or form. A keyboard user activating the skip link is deposited at the banner expand control, not at the wizard question, form field, or review section. This means keyboard users must Tab through the full header, navigation, and banner before reaching any functional content on every page load and on every step transition.

skip link text: "Here's how you know" skip link href: "#collapseFlag" target element: banner collapse toggle (not main content) distance to first form control after header: ~8 Tab stops

WCAG 2.4.1 Bypass Blocks

Recommended FixAdd id="main-content" to the main content container on each page. Change the skip link to <a href="#main-content" class="skip-link">Skip to main content</a>. The existing "Here's how you know" link can remain as a secondary skip target but should not be the primary one.
Serious
Phone input is type="text" — should be type="tel" /form/main

The phone number input (id="rayphone") uses type="text" with no inputmode attribute. Mobile users are presented with a full QWERTY keyboard instead of the numeric dial-pad keyboard. Voice-input and switch-access users lose the semantic hint that triggers phone-appropriate input handling. Browser autofill also relies on input type to correctly identify and populate phone fields.

id: rayphone | type: text | inputmode: null | autocomplete: null Expected: type: tel | autocomplete: "tel"

WCAG 1.3.5 Identify Input Purpose

Recommended FixChange to type="tel" and add autocomplete="tel". Optionally add inputmode="tel" for maximum cross-browser compatibility.
Serious
External links open in new tab with no warning All pages

Four links across the site open in a new browser tab using target="_blank" but provide no visual icon or accessible text warning of this behavior: identitytheft.gov, donotcall.gov, consumer.ftc.gov, and the Privacy Policy. Users of screen readers, users with cognitive disabilities, and users with limited technical proficiency are not informed that activating these links will open a new tab, which constitutes an unexpected change of context.

target="_blank" links: 4 aria-label with "new tab" or "new window" warning: 0 Visible icon indicating new tab: 0

WCAG 3.2.2 On Input · G201 Giving users advanced warning when opening a new window

Recommended FixFor each external link, add either: (a) visually hidden text <span class="sr-only">(opens in new tab)</span> appended inside the link, or (b) aria-label="Identity Theft Resources (opens in new tab)". Additionally add a visible external-link icon with aria-hidden="true".
Serious
Focus drops to BODY on page navigation — not managed All page transitions

When advancing between steps (wizard → form → confirmation), focus is not managed. After each page transition, document.activeElement resolves to BODY — the page background container. Screen reader users receive no announcement of the new page context and must navigate from the top of the document to find the new content. This is a Single Page Application (Angular) pattern where the framework handles routing without browser-native focus restoration, and no manual focus management compensates.

After wizard → /form/main: activeElement = BODY# After /form/main → /form/confirmation: activeElement = BODY# Expected: focus should move to H1 or main landmark on each transition

WCAG 2.4.3 Focus Order · 3.2.2 On Input

Recommended FixOn Angular route change events, programmatically move focus to the page's H1 (once added) or to the main landmark: document.querySelector('h1, main, [role="main"]').focus();. The target element needs tabindex="-1" to receive programmatic focus without appearing in the tab order.

Moderate Violations

Moderate
Email input is type="text" — should be type="email" /form/main

The email input (id="rayemail") uses type="text". Mobile users see a standard keyboard instead of one with the @ symbol prominently displayed. Browser-native email format validation is bypassed. Autofill cannot reliably identify the field as an email address without the correct type.

id: rayemail | type: text | autocomplete: null Expected: type: email | autocomplete: "email"

WCAG 1.3.5 Identify Input Purpose

Recommended FixChange to type="email" and add autocomplete="email".
Moderate
Progress bar missing aria-valuemin attribute All pages

The progress bar uses role="progressbar" and has aria-valuenow and aria-valuemax, but is missing aria-valuemin. The ARIA spec requires all three attributes for a determinate progressbar to be fully machine-readable. Without aria-valuemin, some screen readers cannot correctly calculate the percentage or range of progress.

aria-label: "75% of 100% progress bar" (static label, not dynamic) aria-valuenow: "75" aria-valuemax: "100" aria-valuemin: null ← missing

WCAG 4.1.2 Name, Role, Value · ARIA 1.1 progressbar pattern

Recommended FixAdd aria-valuemin="0". Also update aria-label dynamically on each step change rather than embedding the percentage as a static string.
Moderate
"Leave this page?" dialog uses H5 — inconsistent and incorrect heading level /form/main

All other dialogs in the application use H3 for their heading. The "Leave this page?" confirmation dialog uses H5, breaking the heading hierarchy within the modal and making it inconsistent with the rest of the application. Screen reader users navigating by headings within a dialog expect a consistent heading level. The H5 would also be subordinate to any H3 content behind the modal in the accessibility tree.

All other dialogs: heading level H3 "Leave this page?" dialog: heading level H5 Expected: H3 (consistent with all other dialogs)

WCAG 1.3.1 Info and Relationships · 2.4.6 Headings and Labels

Recommended FixChange the "Leave this page?" dialog heading from <h5> to <h3> to match all other modal headings in the application.
Moderate
No prefers-reduced-motion support detected All pages

No @media (prefers-reduced-motion: reduce) CSS query was detected in any of the site's stylesheets. The Angular step transitions and modal slide-in animations play unconditionally regardless of a user's OS-level reduced-motion preference. This can cause discomfort, disorientation, or vestibular disturbance for users with motion sensitivity, including users with inner ear conditions, vestibular disorders, or epilepsy risk.

prefers-reduced-motion media query: not found in any stylesheet

WCAG 2.3.3 Animation from Interactions (WCAG 2.1 AAA, best practice at AA)

Recommended FixAdd a stylesheet block: @media (prefers-reduced-motion: reduce) { *, *::before, *::after { animation: none !important; transition: none !important; } }. This is a single addition that eliminates the entire category of motion-related complaints.

Remediation Priority & Effort Estimate

The following groups issues by remediation urgency and estimated development effort.

Priority 1 — Immediate (1–3 days dev)

Stop blocking users from completing the form

Add focus outline restoration (1 line of CSS removed). Add type="tel" and type="email" to phone and email fields. Add autocomplete tokens to all personal fields. Add aria-valuemin to progressbar. These are attribute-level fixes with no architectural impact.

Priority 2 — Short-term (1–2 sprint cycles)

Fix structural ARIA failures

Add H1 to all pages. Wrap all radio groups in fieldset+legend. Fix skip link target. Add external link new-tab warnings. Correct "Leave this page?" H5 to H3. Add prefers-reduced-motion stylesheet block.

Priority 3 — Medium-term (requires Angular work)

Fix SPA focus management

Implement focus management on Angular route transitions. Implement focus movement on dynamic section injection (subcategory appearance). Implement focus return to first error field on validation failure.

Priority 4 — Policy (non-code)

Add accessibility contact

Add Section 508 accessibility link and contact email to the footer of reportfraud.ftc.gov. Link to ftc.gov/accessibility. This requires a footer template update and potentially a policy decision on a dedicated accessibility contact for this subdomain.

A Note on Automated Accessibility Tools

Browser extension tools such as axe DevTools, WAVE, and Lighthouse reported no critical violations on these pages. This is consistent with published research indicating that automated tools detect approximately 30–40% of WCAG failures. The failures documented in this report — missing fieldset/legend context, focus management failures, dynamic content injection without announcement, input type mismatches, and autocomplete omissions — are categories that require manual evaluation and cannot be reliably detected by DOM-scanning automation. A passing automated accessibility scan does not indicate WCAG conformance.
